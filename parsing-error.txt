## Hypotheses to Investigate

### 1. **Multiple `[Captures]` patterns**
Both failing files have `[Captures]` sections with multiple regex patterns. This may confuse the parser when transitioning to `[Asserts]`.

**06-adicionais.clurl:**
- Has 2 `[Captures]` blocks in the file
- Line 66: `disp_adicional_uuid: stdout regex /"uuid":\s*"([0-9a-f-]{36})"/`
- Line 70: First assert after capture → fails

### 2. **Regex escape character handling**
The capture regex contains `\s*` which includes a backslash. If the parser doesn't properly handle escape sequences in capture patterns, it may misinterpret subsequent lines.

### 3. **Section transition parsing**
The error occurs specifically on the **first line of `[Asserts]`** immediately following a `[Captures]` block. This suggests the parser may not properly finalize the captures section before starting asserts.

### 4. **Line counting offset**
Line 70 in 06-adicionais.clurl and line 118 in 15-funcionarios.clurl may not correspond to the actual parser's internal line counter if there's an off-by-one error or if blank lines/comments are counted differently.

## Suggested Fixes for clurl Parser

1. **Add debug logging** to show which token/character causes the "invalid assert" error
2. **Verify section transition logic** - ensure `[Captures]` → `[Asserts]` transitions are handled correctly
3. **Test with multiple capture patterns** followed by asserts
4. **Validate regex escape handling** - ensure `\s*` and similar patterns don't leak into next section parsing
5. **Add syntax validation mode** - `clurl --validate <file>` to check syntax without executing
